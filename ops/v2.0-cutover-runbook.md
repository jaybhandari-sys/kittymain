# v2.0.0 Cutover Runbook (per-camera)

This runbook describes the one-shot v1.13.x → v2.0.0 migration for a single
camera in the kitty-augentix fleet. Run it on each camera in turn; there is
no on-device migration logic — the translation is operator-driven.

After all cameras are on v2.0.0, the cutover script (`ops/cutover-v2.0.sh`)
and this runbook can be deleted from the repo.

## Architecture notes (v2.0 vs v1.13.x)

v2.0 has **no on-device persistent runtime-config file**.  The 17-key payload
(NODE_ID / CTRL_* / EDGE_* / RELAY_* / SRC_* / LATENCY_MS / VERBOSE) is
delivered to the camera as an **MQTT retained message** on topic
`torque/rx/<SERIAL>/81`.  The broker stores the payload once; every time the
camera subscribes (boot, reconnect, launcher respawn), the broker auto-
delivers it.  ap2p applies it to in-memory state only.

So "config rotation" for v2.0 is just `mosquitto_pub --retain -t … -f …`.
The change reaches every subscribing camera within seconds.  No per-camera
deploy script needed once a camera is on v2.0.

The on-camera footprint after a successful cutover is **five items**:

```
/mny/mtd/ipc/ambicam/
├── ambicam.sh
├── ap2p
├── config.json     (MQTT broker URL + creds + NODE_ID — bootstrap only)
├── VERSION
└── LICENSES/
```

No `ap2p.conf`, no `provider_srt.conf`, no `.so` files of ours.

## 1. Preconditions

Before starting on a given camera, confirm:

- [ ] Camera is currently running **v1.13.2**
      (`cat /mny/mtd/ipc/ambicam/VERSION` → `1.13.2`).
- [ ] Camera has a pre-cutover backup at
      `/mnt/tf/kitty-backup-pre-v1.13.0/` (created during the v1.13.x deploy).
- [ ] Operator workstation has:
  - `expect` (`brew install expect` / `apt install expect`)
  - `mosquitto-clients` (`brew install mosquitto` / `apt install mosquitto-clients`)
  - `sed`, `grep` (standard)
- [ ] Operator has SSH access to the GCP staging VM
      (`kitty-signaling-server-del`, 34.131.37.52) for fetching the release
      tarball if it isn't already mirrored locally.
- [ ] Operator workstation has network reach to the camera (telnet/22).
- [ ] Camera serial number is known and matches the format
      `ATPL-NNNNNN-XXXX` (e.g. `ATPL-200007-TESTA`).

## 2. Steps

For each camera, follow these steps in order. The full cycle is ~3 minutes
per camera if nothing goes wrong.

1.  **Fetch the v2.0.0 release tarball** from GitHub Releases:

    ```sh
    cd /tmp
    curl -L -o kitty-v2.0.0.tar.gz \
        https://github.com/Adiance-Technologies/kitty-augentix-camera/releases/download/v2.0.0/kitty-v2.0.0.tar.gz
    tar tzf kitty-v2.0.0.tar.gz | head     # sanity check
    ```

    Expected contents: `ap2p`, `ambicam.sh`, `VERSION`, `LICENSES/`.
    (No `ap2p.conf` — runtime config is delivered over MQTT, not packaged.)

2.  **Translate this camera's config**:

    ```sh
    cd kitty-augentix-camera
    ./ops/cutover-v2.0.sh ATPL-200007-TESTA
    ```

    This fetches the v1 conf via telnet, runs the 17-key rename table, and
    writes `/tmp/v2.conf.ATPL-200007-TESTA`.

3.  **Review the translated conf** before publishing:

    ```sh
    cat /tmp/v2.conf.ATPL-200007-TESTA
    ```

    Confirm: 17 keys present (`NODE_ID`, `CTRL_HOST`, `CTRL_PORT`,
    `CTRL_SCHEME`, `CTRL_TOKEN`, `EDGE_HOST`, `EDGE_PORT`, `RELAY_HOST`,
    `RELAY_PORT`, `RELAY_USER`, `RELAY_PASS`, `SRC_HOST`, `SRC_PORT`,
    `SRC_PATH`, `SRC_AUTH`, `LATENCY_MS`, `VERBOSE`), no stale v1 keys.

4.  **Publish (retained) via MQTT** — broker stores the payload and auto-
    delivers it to any subscriber forever (until you re-publish or clear it):

    ```sh
    ./ops/cutover-v2.0.sh ATPL-200007-TESTA --push-via-mqtt
    ```

    Under the hood this is `mosquitto_pub --retain …`.  Even if the camera
    isn't connected right now, the next time it subscribes the broker hands
    it this payload immediately.

5.  **Replace the on-camera binaries** (operator-driven, see
    `ops/camera-deploy-guide.md` for the v1.13.x deploy pattern adapted to
    v2.0 — same telnet+`wget` flow, but you push only `ap2p` and
    `ambicam.sh`; there is no `ap2p.conf` to push).

6.  **Wait ~10 seconds** for the new launcher to start `ap2p`, the MQTT
    thread to subscribe, and the broker to deliver the retained payload.

7.  **Telnet to the camera and verify the v2.0 filesystem layout**:

    ```sh
    telnet 192.168.12.129
    # login: root  /  pwd: adiance@999@arcisai
    ls -la /mny/mtd/ipc/ambicam/
    ```

    Expected — **five items only**:
    - `ambicam.sh`, `ap2p`, `config.json`, `VERSION`, `LICENSES/`.
    - **No** `ap2p.conf` (it never lives on disk in v2.0 — runtime config
      arrives via MQTT retained message).
    - **No** `provider_srt*` files, **no** `MQTT_vcamclient*`, **no**
      `P2Pambicam*`, **no** loose `lib*.so*` files.

8.  **Reboot the camera** (preferred) or kill + restart the launcher
    in-place:

    ```sh
    # in-place:
    killall ambicam.sh; killall ap2p; killall provider_srt 2>/dev/null
    /mny/mtd/ipc/ambicam/ambicam.sh &
    # or just:
    reboot
    ```

9.  **Verify `ap2p` is running**:

    ```sh
    ps w | grep -E 'ap2p|ambicam' | grep -v grep
    cat /mny/mtd/ipc/ambicam/VERSION    # should now read 2.0.0
    tail /mny/mtd/ipc/ambicam/ap2p.log
    ```

## 3. Verification checklist

Run on the camera (via telnet) after step 8:

```sh
# (a) Filesystem layout — exactly the v2.0 set, no leftovers.
ls -la /mny/mtd/ipc/ambicam/

# (b) Version stamp.
cat /mny/mtd/ipc/ambicam/VERSION

# (c) ap2p process up.
ps w | grep ap2p | grep -v grep

# (d) Launcher process up.
ps w | grep ambicam.sh | grep -v grep

# (e) ap2p log producing output (last 20 lines).
tail -20 /mny/mtd/ipc/ambicam/ap2p.log

# (f) No legacy daemons left running.
ps w | grep -E 'provider_srt|MQTT_vcamclient|P2Pambicam' | grep -v grep
# (expect: empty)

# (g) Free space healthy.
df -h /mny/mtd/ipc
```

All seven checks should pass. The expected-empty check (f) is the one most
likely to surface a half-migration — if anything shows up there, jump to
Rollback.

---

## 4. Rollback (v2.0 failed → restore v1.13.2)

The v1.13.0 deploy left a complete backup of the pre-cutover state at
`/mnt/tf/kitty-backup-pre-v1.13.0/`. To restore on a single camera:

```sh
telnet 192.168.12.129    # root / adiance@999@arcisai

# 1. Stop the v2.0 launcher and binary.
killall ambicam.sh 2>/dev/null
killall ap2p 2>/dev/null

# 2. Clear the v2.0 directory.
rm -f /mny/mtd/ipc/ambicam/ap2p \
      /mny/mtd/ipc/ambicam/ap2p.log \
      /mny/mtd/ipc/ambicam/VERSION
rm -rf /mny/mtd/ipc/ambicam/LICENSES

# 3. Restore the v1.13.2 fileset from the SD-card backup.
cp -a /mnt/tf/kitty-backup-pre-v1.13.0/. /mny/mtd/ipc/ambicam/

# 4. Confirm version.
cat /mny/mtd/ipc/ambicam/VERSION   # should read 1.13.2

# 5. Reboot to bring up the restored v1 launcher cleanly.
reboot
```

After reboot, run the v1.13.2 verification checks from
`ops/dev-integration-guide.md` to confirm the legacy stack is healthy
(MQTT client, provider_srt, log files growing).

If the backup directory is missing or corrupt, escalate before doing
anything else — do **not** improvise a partial restore.
